: Decorator Pattern
You need to add functionality to an object at runtime, but subclassing would lead to an explosion of subclasses or is impractical.
: Decorator Pattern
Question :- Let's say we have a simple Pizza ordering system. Initially, we just have plain pizza but now we want to add options such as 
cheese, olives, tomatoes and mushrooms without modifying the existing pizza class or creating multiple subclasses.
Let's use the Decorator Pattern using a Pizza making example where different toppings (like cheese, olives, tomatoes, etc.) 
can be dynamically added to the base pizza.

Decorator Pattern
Definition
Decorator Pattern allows you to add new behavior or functionality to an object dynamically without modifying its existing code.
Or simply:
Wrap an object with another object to add extra features.


Why do we need it?
Suppose you're building a Pizza ordering system.
Initially, you only have:
Plain Pizza
Later, customers want:
Cheese
Olives
Mushrooms
Tomatoes
Without the Decorator Pattern, you might create subclasses like: as topping increse subclass increse exponsily 

This is hard to maintain.
Instead, keep one base pizza and decorate it with toppings.


Advantages
Add functionality without modifying existing classes.
Avoids creating many subclasses.
Supports combining features dynamically.
Follows the Open/Closed Principle.
Flexible and reusable.

Disadvantages
Introduces many small decorator classes.
The wrapping order can matter in some cases.
Can make debugging harder because of multiple wrapper layers.
